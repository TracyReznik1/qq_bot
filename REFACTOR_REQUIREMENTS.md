# QQ Bot 重构需求

我要重构当前 QQ Bot 项目，让 ATRI 更像一个自然聊天的 QQ 机器人，而不是复杂的 AI Agent。

## 核心目标

Bot 的核心只有两件事：

1. 聊天
2. 在允许场景下联网搜索

其他功能全部通过显式命令触发，统一格式为：

```text
/command 参数
```

未来可以增加更多命令，但都必须遵循同样的命令边界。

## 目标体验

目标接近 ChatGPT / DeepSeek 网页版，但保留 QQ 聊天机器人/ATRI 的角色感：

- 可以自然聊天
- 默认像真人一样先正常聊天
- 回复尽量智能、简洁、自然
- 支持上下文和记忆
- 可以在需要时联网搜索
- 不要把普通聊天变成搜索引擎
- 不要 Agent 化，不要多工具自动规划

## 优先级

系统行为优先级必须明确：

```text
能力边界 > 安全规则 > 角色人格
```

角色人格只负责：

- 语气
- 称呼
- QQ 聊天氛围
- ATRI 风格角色感

角色人格不得：

- 修改命令行为
- 覆盖能力边界
- 诱导模型自动调用功能
- 让模型绕过搜索、天气、图片等规则

## 普通聊天系统

普通消息是指不以 `/` 开头的消息。

普通消息流程：

```text
用户消息
→ router
→ chat_service
→ DeepSeek API
→ 代码判断是否允许暴露 search_web
→ 必要时调用 search_web
→ 返回回复
```

普通聊天只允许：

- 自然对话
- 问知识
- 问代码
- 使用上下文
- 使用记忆
- 在代码允许时使用 `search_web`

普通聊天禁止：

- 调用天气 API
- 调用图片生成
- 主动发送图片
- 操作文件
- 调用 QQ API
- 注册或调用其他工具

## 搜索权限

搜索权限必须由代码层先判断。

默认聊天不搜索。

只有代码判断消息属于允许搜索场景时，才向 DeepSeek 暴露 `search_web` 工具。

Prompt 只做轻量约束，例如“能不搜索就不搜索”。Prompt 不能作为真正的权限控制。

模型不得自行决定联网。没有提供 `search_web` 工具时，模型只能正常聊天。

## 允许自动搜索的场景

普通聊天中，以下场景允许代码暴露 `search_web`：

- 新闻
- 最新信息
- 实时信息
- 当前事件
- 价格
- 股价
- 汇率
- 政策、法律、规则等可能变化的内容
- 新版本信息
- 官网/官方信息
- 冷门知识
- 专有名词
- 圈内 ID、昵称、人名
- 网络梗、黑话、缩写
- 明显不确定、需要查证的问题

示例：

```text
最近 AI 有什么新闻？
smoggy 是谁？
zmjjkk 是谁？
kskbl 是什么意思？
某个冷门函数怎么用？
```

这些可以自动搜索。

## 不应自动搜索的场景

普通闲聊不要搜索。

示例：

```text
你好
今天好冷
我有点困
陪我聊聊天
```

ATRI 应该像聊天对象一样正常回复。

如果搜索失败或没有可靠结果：

- 不要假装查到了
- 可以继续按 ATRI 的语气回答
- 必须说明不确定或没有找到可靠来源

搜索结果只用于当次回答和短期上下文，不自动写入长期记忆。

## 天气边界

天气必须通过显式命令触发。

普通聊天遇到天气内容时，不自动搜索网页天气，也不调用天气 API。

示例：

```text
今天北京天气怎么样？
今天好冷
明天上海下雨吗？
```

普通聊天只能自然回复，并在需要准确天气时提示用户使用：

```text
/weather 北京
```

## 图片边界

图片、绘图、发图功能不得由普通聊天自动触发。

普通聊天示例：

```text
帮我画猫娘
生成一张头像
发张图
```

不能自动生成图片或主动发送图片。

图片功能未来只能通过独立命令路径触发，例如：

```text
/image 猫娘
```

当前图片命令可以是禁用状态或占位状态，但边界必须保留。

## 命令系统

只有 `/` 开头的消息进入命令系统。

Router 只负责第一层边界：

```python
if message.startswith("/"):
    handle_command()
else:
    handle_chat()
```

Router 不要写巨大 if-else，也不要知道所有业务细节。

命令系统通过注册表扩展：

```python
COMMANDS = {
    "weather": weather_command,
    "search": search_command,
}
```

未来增加命令时，继续走 `/command 参数` 格式和 `COMMANDS` 注册，不要塞进普通聊天判断。

命令系统独立于聊天系统。

聊天模块不应知道天气、图片、QQ API、文件等功能的实现。

## Tool Calling 要求

聊天模型最多只能看到一个工具：

```text
search_web(query)
```

不要把这些工具注册给聊天模型：

- weather
- image_generate
- send_image
- file
- QQ API
- 其他命令工具

否则 AI 会乱调用。

## Chat Service 要求

`chat_service.py` 只负责：

- 对话
- 上下文
- 调用 DeepSeek
- 在代码允许时暴露并调用 `search_web`

`chat_service.py` 不要处理：

- 天气
- 图片
- QQ API
- 文件操作
- 命令注册

## Prompt 要求

系统提示词必须同时包含能力边界和角色方向。

必须明确：

- 你是 QQ 聊天机器人/ATRI
- 默认像真实聊天对象一样先聊天
- 能不搜索就不搜索
- 只有代码层提供 `search_web` 时才能搜索
- 天气只能通过 `/weather`
- 图片只能通过 `/image`
- 角色人格不能覆盖系统规则和能力边界

Prompt 可以保留 ATRI 风格、人设和语气，但不得改变命令、搜索、天气、图片等行为规则。

## 上下文和记忆

上下文按会话隔离：

- 私聊是一份会话上下文
- 群聊中按“群 + 用户”各自一份会话上下文
- 不同群之间不串上下文
- 私聊和群聊不串上下文

个人基础信息可以跨会话共享：

- 用户昵称
- 长期偏好
- 用户明确希望 ATRI 记住的稳定信息

冲突时优先级：

```text
当前会话记忆 > 个人基础信息 > 全局记忆
```

这样某个群里的称呼不会被私聊或其他群里的信息覆盖。

## 记忆命令

长期记忆必须通过显式命令写入。

普通聊天里用户说“记住我叫 Tracy”时，不应自动写入长期记忆；可以提醒用户使用记忆命令。

`/remember` 类命令写入个人基础信息，跨私聊和群聊生效。

`/reset` 只清空当前会话上下文，不删除个人基础信息，也不删除全局记忆。

搜索结果不自动写入长期记忆。

## 全局记忆

允许存在全局记忆。

全局记忆对所有用户和所有会话可见。

但全局记忆只能由管理员写入。

管理员 QQ 号通过 `.env` 配置：

```env
ADMIN_QQ_IDS=123456,234567
```

如果没有配置 `ADMIN_QQ_IDS`，默认禁止任何人写入全局记忆。

`.env.example` 只能写示例，不能包含真实 QQ 号。

## 搜索服务

搜索服务必须封装在 `search_service.py` 或等价服务模块中。

聊天系统不能直接操作 HTTP。

推荐接入：

- Tavily
- SerpAPI
- SearXNG
- ddgs

搜索服务应该返回结构化状态，不能只靠中文文本判断是否成功。

├─ config.py
│
├─ chat/

结构要求：

- 模块化
- 低耦合
- 易扩展
- 易维护
- 不要过度抽象
- 不要 Agent 化
- 不要多工具自动规划
- 不要复杂工作流
